Detailed Description

Includes APIs for enumerating cameras, creating handles, and turning on cameras.

Function Documentation

◆ MV_CC_EnumDevices()

MV_CAMCTRL_API int __stdcall MV_CC_EnumDevices ( IN unsigned int  nTLayerType,
IN OUT MV_CC_DEVICE_INFO_LIST *  pstDevList 
)

Enumerates devices, including cameras connected to frame grabbers.

Parameters
nTLayerType[IN] It refers to the transport layer protocol type. For more details, refer to CameraParams.h.
Transport Layer Protocol Type Value Description
MV_GIGE_INTERFACE 0x00000001 GigE vision frame grabber.
MV_CAMERALINK_INTERFACE 0x00000004 Camera link frame grabber.
MV_CXP_INTERFACE 0x00000008 CoaXPress frame grabber.
MV_XOF_INTERFACE 0x00000010 XoFLink frame grabber.
pstDevList[IN][OUT] It refers to the device list.
Returns
Returns MV_OK for success, and returns corresponding Error Code for failure.
Note
 • The memory of device list is internally allocated. When this API is called in multiple threads, the SDK will release and apply for the device list memory. It is recommended to avoid multithreaded enumeration operations.
 • For the parameter nTLayerType, the following parameters are supported: MV_GIGE_DEVICE, MV_1394_DEVICE, MV_USB_DEVICE, and MV_CAMERALINK_DEVICE. MV_GIGE_DEVICE sends out information of all GigE devices (including virtual GigE devices and GigE devices of GenTL), and MV_USB_DEVICE sends out information of USB devices (including virtual USB devices).
 • The transport layers of MV_GENTL_GIGE_DEVICE, MV_GENTL_CAMERALINK_DEVICE, MV_GENTL_CXP_DEVICE, and MV_GENTL_XOF_DEVICE can return the information of camera connected to corresponding frame grabber.
Examples
ChunkData.cpp, Display.cpp, Events.cpp, ForceIPEx.cpp, Grab_Asynchronous.cpp, GrabImage.cpp, GrabImage_HighPerformance.cpp, GrabMultipleCamera.cpp, HighBandwidthDecode.cpp, ImageProcess.cpp, ImageSave.cpp, LineScanSoftwareTrigger.cpp, MultiCast.cpp, MultiLightCtrl_ImageStitching.cpp, ParametrizeCamera_AreaScanIOSettings.cpp, ParametrizeCamera_FileAccess.cpp, ParametrizeCamera_LineScanIOSettings.cpp, ParametrizeCamera_LoadAndSave.cpp, QuickSoftwareTrigger.cpp, ReconnectDemo.cpp, SetParam.cpp, and Trigger_Image.cpp.

◆ MV_CC_EnumDevicesEx()

MV_CAMCTRL_API int __stdcall MV_CC_EnumDevicesEx ( IN unsigned int  nTLayerType,
IN OUT MV_CC_DEVICE_INFO_LIST *  pstDevList,
IN const char *  strManufacturerName 
)

Enumerates devices according to manufacturers, including cameras connected to the frame grabber.

Parameters
nTLayerType[IN] It refers to the transport layer protocol type. For more details, refer to CameraParams.h.
Transport Layer Protocol Type Value Description
MV_GIGE_DEVICE 0x00000001 GigE device.
MV_USB_DEVICE 0x00000004 USB device.
MV_CAMERALINK_DEVICE 0x00000008 Camera link device (serial port).
MV_VIR_GIGE_DEVICE 0x00000010 Virtual GigE device.
MV_VIR_USB_DEVICE 0x00000020 Virtual USB device.
MV_GENTL_GIGE_DEVICE 0x00000040 Self-developed NIC-based GigE device.
MV_GENTL_CAMERALINK_DEVICE 0x00000080 Camera link device connected to camera link frame grabber.
MV_GENTL_CXP_DEVICE 0x00000100 CoaXPress device.
MV_GENTL_XOF_DEVICE 0x00000200 XoF device.
pstDevList[IN][OUT] It refers to the device list.
strManufacturerName[IN] It refers to the manufacturers.
Returns
Returns MV_OK for success, and returns corresponding Error Code for failure.
Note
 • For the parameter nTLayerType, the following parameters are supported: MV_GIGE_DEVICE, MV_1394_DEVICE, MV_USB_DEVICE, and MV_CAMERALINK_DEVICE. MV_GIGE_DEVICE sends out information of all GigE devices (including virtual GigE devices and GigE devices of GenTL), and MV_USB_DEVICE sends out information of USB devices (including virtual USB devices).
 • The memory of device list is internally allocated. When this API is called in multiple threads, the SDK will release and apply for the device list memory. It is recommended to avoid multithreaded enumeration operations.
 • MV_GENTL_GIGE_DEVICE, MV_GENTL_CAMERALINK_DEVICE, MV_GENTL_CXP_DEVICE, and MV_GENTL_XOF_DEVICE can return the camera information under the corresponding transport layer.

◆ MV_CC_EnumDevicesEx2()

MV_CAMCTRL_API int __stdcall MV_CC_EnumDevicesEx2 ( IN unsigned int  nTLayerType,
IN OUT MV_CC_DEVICE_INFO_LIST *  pstDevList,
IN const char *  strManufacturerName,
IN MV_SORT_METHOD  enSortMethod 
)

Enumerates devices, supporting enumerating devices by specified sorting method and filtering by manufacturer name.

Parameters
nTLayerType[IN] It refers to the transport layer protocol type. For more details, refer to CameraParams.h.
Transport Layer Protocol Type Value Description
MV_GIGE_DEVICE 0x00000001 GigE device.
MV_USB_DEVICE 0x00000004 USB device.
MV_CAMERALINK_DEVICE 0x00000008 Camera link device (serial port).
MV_VIR_GIGE_DEVICE 0x00000010 Virtual GigE device.
MV_VIR_USB_DEVICE 0x00000020 Virtual USB device.
MV_GENTL_GIGE_DEVICE 0x00000040 Self-developed NIC-based GigE device.
MV_GENTL_CAMERALINK_DEVICE 0x00000080 Camera link device.
MV_GENTL_CXP_DEVICE 0x00000100 CoaXPress device.
MV_GENTL_XOF_DEVICE 0x00000200 XoF device.
pstDevList[IN][OUT] It refers to the device list.
strManufacturerName[IN] It refers to the name of the manufacturer (NULL means not filtering).
enSortMethod[IN] It refers to the sorting type.
Returns
Returns MV_OK for success, and returns corresponding Error Code for failure.
Note
 • The memory of device list is internally allocated. When this API is called in multiple threads, the SDK will release and apply for the device list memory. It is recommended to avoid multithreaded enumeration operations.
 • strManufacturerName can be set to NULL, which indicates enumerating all devices according to the specified sorting type; if not set to NULL, the sorted device list of specified manufacturers will be returned.
 • MV_GENTL_GIGE_DEVICE, MV_GENTL_CAMERALINK_DEVICE, MV_GENTL_CXP_DEVICE, and MV_GENTL_XOF_DEVICE can return the camera information under the corresponding transport layer.
 • The differences between MV_CC_EnumDevicesEx2() and MV_CC_EnumDevicesEx() are as follows:
  • If MV_GIGE_DEVICE is configured for MV_CC_EnumDevicesEx(), virtual GigE cameras and GigE cameras connected to self-developed frame grabbers will be enumerated besides the GigE cameras in the network segment; if MV_USB_DEVICE is configured, USB cameras and virtual USB cameras will be enumerated.
  • If MV_GIGE_DEVICE is configured for MV_CC_EnumDevicesEx2(), only GigE cameras in the network segment will be enumerated; if MV_USB_DEVICE is configured, only USB cameras will be enumerated.
  • MV_CC_EnumDevicesEx2() contains an extra sorting feature. The sorting type is determined by the parameter MV_SORT_METHOD enSortMethod.

◆ MV_CC_IsDeviceAccessible()

MV_CAMCTRL_API bool __stdcall MV_CC_IsDeviceAccessible ( IN MV_CC_DEVICE_INFO *  pstDevInfo,
IN unsigned int  nAccessMode 
)

Checks if the specified device can be accessed.

Parameters
pstDevInfo[IN] It refers to device information structure.
nAccessMode[IN] It refers to access mode.
Macro Value Description
MV_ACCESS_Exclusive 1 The device owns exclusive permission. For other apps, only the CCP register is allowed to be read.
MV_ACCESS_ExclusiveWithSwitch 2 The device can be preemptively accessed under mode 5 and then turned on with exclusive permission.
MV_ACCESS_Control 3 The device can be turned on with control permission. For other apps, all registers are allowed to be read.
MV_ACCESS_ControlWithSwitch 4 The device can be preemptively accessed under mode 5 and then turned on with control permission.
MV_ACCESS_ControlSwitchEnable 5 The device can be turned on with preemptible control permission.
MV_ACCESS_ControlSwitchEnableWithKey 6 The device can be preemptively accessed under mode 5 and then turned on with preemptible control permission.
MV_ACCESS_Monitor 7 The device can be turned on in read mode. It is applicable to devices under control permission.
Returns
Returns true for accessible status, and false for inaccessible status.
Note
 • You can read the device CCP register value to check the current access permission.
 • Return false if the device does not support the modes MV_ACCESS_ExclusiveWithSwitch, MV_ACCESS_ControlWithSwitch, and MV_ACCESS_ControlSwitchEnableWithKey. Currently, the device does not support the 3 modes, neither do the devices from other mainstream manufacturers.
 • This API is not supported by camera link devices
Examples
ForceIPEx.cpp.

◆ MV_CC_OpenDevice()

MV_CAMCTRL_API int __stdcall MV_CC_OpenDevice ( IN void *  handle,
IN unsigned int  nAccessMode,
IN unsigned short  nSwitchoverKey 
)

Turns on the device.

Parameters
handle[IN] It refers to the device handle.
nAccessMode[IN] It refers to access mode.
Macro Value Description
MV_ACCESS_Exclusive 1 The device owns exclusive permission. For other apps, only the CCP register is allowed to be read.
MV_ACCESS_ExclusiveWithSwitch 2 The device can be preemptively accessed under mode 5 and then turned on with exclusive permission.
MV_ACCESS_Control 3 The device can be turned on with control permission. For other apps, all registers are allowed to be read.
MV_ACCESS_ControlWithSwitch 4 The device can be preemptively accessed under mode 5 and then turned on with control permission.
MV_ACCESS_ControlSwitchEnable 5 The device can be turned on with preemptible control permission.
MV_ACCESS_ControlSwitchEnableWithKey 6 The device can be preemptively accessed under mode 5 and then turned on with preemptible control permission.
MV_ACCESS_Monitor 7 The device can be turned on in read mode. It is applicable to devices under control permission.
nSwitchoverKey[IN] It refers to the secret key for switching access permission.
Returns
Returns MV_OK for success, and returns corresponding Error Code for failure.
Note
 • You can find the device and connect it according to the configured device parameters.
 • When the API is called, the parameters nAccessMode and nSwitchoverKey are optional, and the device access mode is exclusive by default. Currently the device does not support the following preemption modes: MV_ACCESS_ExclusiveWithSwitch, MV_ACCESS_ControlWithSwitch, MV_ACCESS_ControlSwitchEnable, and MV_ACCESS_ControlSwitchEnableWithKey.
 • For USB3 vision device, the parameters nAccessMode and nSwitchoverKey are invalid.
Examples

◆ MV_CC_GetAllMatchInfo()

MV_CAMCTRL_API int __stdcall MV_CC_GetAllMatchInfo ( IN void *  handle,
IN OUT MV_ALL_MATCH_INFO *  pstInfo 
)

Gets information of all types.

Parameters
handle[IN] It refers to the device handle.
pstInfo[IN][OUT] It refers to the pointer to information structure.
Returns
Returns MV_OK for success, and returns corresponding Error Code for failure.
Note
 • Input required information type (specify nType in structure MV_ALL_MATCH_INFO ), and get corresponding information (returned via pInfo in structure MV_ALL_MATCH_INFO ).
 • The calling precondition of this API is determined by the required information type. To obtain MV_MATCH_TYPE_NET_DETECT information of GigE devices, this API should be called after image grabbing starts. To obtain MV_MATCH_TYPE_USB_DETECT information of U3V devices, this API should be called after the device is turned on.
 • This API is not supported by camera link devices.

◆ MV_CC_GetDeviceInfo()

MV_CAMCTRL_API int __stdcall MV_CC_GetDeviceInfo ( IN void *  handle,
IN OUT MV_CC_DEVICE_INFO *  pstDevInfo 
)

Gets device information, used before image grabbing.

Parameters
handle[IN] It refers to the device handle.
pstDevInfo[IN][OUT] It refers to the pointer to device information structure.
Returns
Returns MV_OK for success, and returns corresponding Error Code for failure.
Note
 • Call this API after MV_CC_OpenDevice() and before MV_CC_StartGrabbing() to get the device information.
 • The API is not supported by GenTL devices.
 • For a GigE device, there is a blocking risk when calling the API, so it is not recommended to call the API during image grabbing.
